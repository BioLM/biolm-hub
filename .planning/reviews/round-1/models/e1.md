# Review — `models/e1/` (Round 1)

**Target:** `models/e1/` — Profluent/Synthyra **E1**, a retrieval-augmented protein
**encoder** language model (variants 150m/300m/600m; actions `encode`, `predict`, `log_prob`).

## Summary

E1 is a well-built, house-conforming model. Layout is complete (all standard code files +
the 5-file knowledge graph + `LICENSE`), it uses the canonical actions (`encode`/`predict`/
`log_prob`), self-populates weights via `r2_then_hf` with a pinned HF revision per variant,
follows the pinned `field_glossary.yaml` wording for `logits`/`vocab_tokens`/`layer`/
`per_token_embeddings`/`log_prob`, uses `get_logger` (no `print`), and ships lazy-loading
integration + deployment tests that reuse the shared `STANDARD_PROTEIN` asset. The custom
parts (multi-sequence/block-causal context input, runtime `auto_map` patch for
`trust_remote_code`, dtype-by-GPU selection, snapshot/torch.compile disables) are the
"science," and they are clearly documented. The trust-remote-code concern is handled
correctly: `r2_then_hf` downloads and re-caches the *full* HF snapshot (so `modeling_e1.py`
and tokenizer files are present on both fallback and R2-cache-hit paths).

The findings are dominated by **documentation accuracy** (not plumbing): the single largest
issue is that the docs repeatedly call E1 an "**Encrypted** protein language model" — it is an
**Encoder** model (paper: "E1: Retrieval-Augmented Protein *Encoder* Models"). There is one
real software-engineering deviation: `encode` **silently drops** out-of-bounds `repr_layers`
instead of raising a typed error the way `esm2` does. No secrets, no licensing problems, no
correctness bug in the math path that the regression fixtures cover. Several findings are
shared house-wide with `esm2` (the reference model) and are flagged as such for global dedupe.

DoD (per-model scope): canonical actions ✓, self-populating acquisition ✓, 5-file KG present ✓,
per-model `LICENSE` (Apache-2.0) consistent with `sources.yaml` ✓, typed schema + logging ✓,
integration+deployment tests ✓. Caveats: doc-accuracy items below, an error-handling gap, and
a house-wide internal `qa` reference in a usage docstring.

## 🟠 Should-fix

### 1. Docs call E1 "Encrypted" — it is an **Encoder** model (factual error, user-facing)
**Category:** Docs / accuracy
**Location:** `models/e1/README.md:3`, `models/e1/MODEL.md:6`, `models/e1/sources.yaml:6`
**Detail:** The README one-line summary (the first thing a contributor/user reads) says
"**Encrypted** protein language model with retrieval-augmented inference"; MODEL.md says
"E1 is an **encrypted** protein language model developed by Profluent Bio and Synthyra";
sources.yaml `notes` says "**Encrypted** protein embedding model by Profluent/Synthyra".
E1 has nothing to do with encryption — the paper is *"E1: Retrieval-Augmented Protein
**Encoder** Models"* and the HF repos are `Synthyra/Profluent-E1-*`. This is a prominent,
repeated factual error in shipped public docs.
**Fix:** Replace "encrypted" with "encoder" (or simply "protein language model") in all three
places, e.g. README one-liner → "Retrieval-augmented protein **encoder** language model that
produces embeddings, masked predictions, and log-probability scores…".

### 2. `encode` silently drops out-of-bounds `repr_layers` instead of raising (deviates from `esm2`)
**Category:** SE / error handling / cross-model consistency
**Location:** `models/e1/app.py:336-340`
**Detail:** Requested layers are filtered with `if 0 <= pos_lyr < n_layers: layers_to_use.append(...)`
— any out-of-range layer is silently discarded. A user who requests `repr_layers=[25]` on the
150m variant (21 hidden states) gets an empty `embeddings: []` with no error or explanation.
The house pattern (`esm2/app.py:244-248`) validates layer bounds at runtime and raises a typed
`ValidationError400` ("layer indices are out of bounds…"). This is both a UX footgun and a
uniformity break (the north star is that error handling is the same across models).
**Fix:** After computing `n_layers`, validate `requested_layers` against
`[-n_layers, n_layers-1]` and raise the same typed `ValidationError400`/`UserError` `esm2`
uses when any requested layer is out of range, instead of silently filtering.

## 🟡 Nits / polish

### 3. MODEL.md says "5 overhead tokens per sequence" but the code uses 4 (`len + 4`)
**Category:** Docs / code mismatch
**Location:** `models/e1/MODEL.md:54` vs `models/e1/app.py:254-257`
**Detail:** MODEL.md states the per-sequence tokenization `<bos> 1 SEQUENCE 2 <eos>` has
"(5 overhead tokens per sequence)". The position math in `_get_query_token_positions` counts
exactly 4 special tokens (`<bos>`, `1`, `2`, `<eos>`): `context_token_count = sum(len(s) + 4 …)`
and `query_start = … + 2`. The code (which the golden fixtures validate) is authoritative; the
doc count of 5 is wrong.
**Fix:** Change "5 overhead tokens" → "4 overhead tokens (`<bos>`, `1`, `2`, `<eos>`)".

### 4. BIOLOGY.md dates E1 to "(2024)"; everywhere else it's 2025
**Category:** Docs / internal consistency
**Location:** `models/e1/BIOLOGY.md:66`
**Detail:** "E1 is a recent model (2024)." The bioRxiv DOI in `sources.yaml`
(`10.1101/2025.11.12.688125`, year 2025) and `comparison.yaml:18` ("Recent model (2025)")
both say 2025. The 2024 is wrong/inconsistent.
**Fix:** "(2024)" → "(2025)".

### 5. MODEL.md training-data table says "details not published" but names the datasets two paragraphs up
**Category:** Docs / internal consistency
**Location:** `models/e1/MODEL.md:36` vs `models/e1/MODEL.md:21`
**Detail:** The "Training Data" table row reads `Source | Protein sequence databases (details
not published)`, but line 21 explicitly states training used "the Profluent Protein Atlas
(PPA-1) and UniRef Version 2411 datasets" (with mixing ratio and token counts). The two
contradict each other.
**Fix:** Replace "details not published" with the named sources (PPA-1 + UniRef 2411), or drop
the contradictory row.

### 6. README/MODEL test-case tables mislabel encode test #2 as "multiple sequences"
**Category:** Docs / accuracy
**Location:** `models/e1/README.md:199`, `models/e1/MODEL.md:82`
**Detail:** Both tables describe the second encode test as "Multi-sequence encode / Multiple
proteins" and "Encode (mean, multiple sequences)". The actual fixture (`fixture.py:67-79`) is a
**single** `STANDARD_PROTEIN` with `include=[per_token]`, `repr_layers=[15]`. No test in the
suite sends a batch of multiple query sequences in one request.
**Fix:** Relabel test #2 to "Encode (per-token, single sequence, layer 15)" in both files.

### 7. `validate_context_sequences` is copy-pasted across all three request-item classes
**Category:** SE / DRY
**Location:** `models/e1/schema.py:90-104`, `:149-168`, `:210-224`
**Detail:** The three field validators share identical empty-check + max-length-check logic,
differing only in the per-residue validator (`validate_aa_extended` vs adds the `?`-rejection
vs `validate_aa_unambiguous`). This is real duplication in one file.
**Fix:** Extract a module-level helper, e.g.
`def _validate_context_list(v, *, residue_validator, forbid_mask=False)`, and have each
validator call it with the appropriate residue validator / flag.

### 8. Context-homolog sequences are hardcoded and duplicated across `test.py` and `fixture.py`
**Category:** Tests / shared-asset reuse (rubric A.10)
**Location:** `models/e1/test.py:73-74`, `models/e1/fixture.py:95-96`
**Detail:** The query reuses the shared `STANDARD_PROTEIN` asset (good), but the two homolog
context sequences (`MPSSKE…QKL`, `TPSSKEL…QKY`) are inlined verbatim in both files. They are
exactly the standard-sequence duplication the rubric asks to avoid, and keeping two copies in
sync is fragile.
**Fix:** Add a shared constant (e.g. `STANDARD_PROTEIN_HOMOLOGS` in
`models/commons/testing/shared_assets.py`, near `STANDARD_PROTEIN`) and reference it from both
`test.py` and `fixture.py`.

### 9. Stale GPU comment: app.py says "A10G" but the model deploys on L4
**Category:** Docs / code comment
**Location:** `models/e1/app.py:172-176` vs `models/e1/config.py:46-56`
**Detail:** The dtype-selection comment says "A10G (300m/600m) uses bfloat16 … A10G (Ampere)
has native bfloat16", but 300m/600m are configured on **L4** (config.py explicitly notes the
A10G→L4 migration, "27% cheaper than A10G"). The logic is correct; the comment is stale.
**Fix:** Update the comment to reference L4 (Ada Lovelace) and drop the A10G mention.

### 10. `config.json` `auto_map` is patched at runtime on every cold start
**Category:** SE / simplicity (optional)
**Location:** `models/e1/app.py:156-170`
**Detail:** Each container cold-start re-reads, conditionally rewrites, and re-dumps
`config.json` to inject `auto_map`. This works (Modal's runtime FS is writable), but it bakes
a per-cold-start file I/O + write dependency into the hot path for something that never
changes. Doing the patch once in `download.py` (so it lands in the image and the R2 cache)
would be cleaner and removes the runtime write.
**Fix:** Move the `auto_map` injection into `download_model_assets` (post-download, before
caching to R2), or upstream a PR to the HF repo's `config.json`.

### 11. Internal env reference "QA/prod" in the usage docstring (house-wide)
**Category:** Open-source readiness / internal leakage
**Location:** `models/e1/app.py:510`
**Detail:** The `__main__` docstring says "# Force deploy in QA/prod:". The rubric lists the
internal `qa` env as an internal-reference leak. This is shared with the reference model
(`esm2/app.py:484` "Force deploy to \"qa\" or \"main\" environment"), so it is a house-wide
cleanup rather than an e1-specific defect — flagging for the global sweep.
**Fix:** Reword generically (e.g. "Force deploy to the configured Modal environment") repo-wide.

### 12. `context_sequences` field name vs the canonical `msa` input (low-confidence uniformity note)
**Category:** Schema / cross-model uniformity
**Location:** `models/e1/schema.py:84-88` (and the predict/log_prob items)
**Detail:** The rubric lists `msa` as a canonical input field for related/homolog sequences,
and `msa_transformer`/`boltz`/`boltzgen`/`rf3` all use `msa`. E1 introduces a new field name
`context_sequences`. This is **defensible**: `msa` in those models means *aligned*,
equal-length sequences (with gap chars, `max_msa_depth`, alignment validation), whereas E1's
context is explicitly *unaligned* raw homologs of differing lengths — so reusing `msa` would
be semantically misleading. Raising it only so the decision is explicit and recorded.
**Fix:** Either keep `context_sequences` (recommended, given the unaligned semantics) and add a
note to `field_glossary.yaml` documenting it as the canonical name for "unaligned homolog
context," or, if a global owner prefers, expose it as `msa` with `context_sequences` as a
Pydantic alias.

## Cross-checks that passed (for the record)
- **Acquisition / trust_remote_code:** `r2_then_hf` (allow_patterns=None) pulls the full HF
  snapshot and re-caches the whole `target_dir` to R2, so `modeling_e1.py` + tokenizer files
  are present on both fallback and cache-hit paths. `huggingface_hub` is correctly listed in
  `setup_download_layer(extra_pip_packages=…)` (build-order rule honored). Revisions pinned
  per variant in `config.py`.
- **Field descriptions / glossary:** `logits`, `vocab_tokens`, `layer`, `per_token_embeddings`,
  and `log_prob` ("Pseudo-log-likelihood…", correct for a masked-LM) match the pinned
  `tooling/field_glossary.yaml` verbatim wording. (The "over the model vocabulary" wording for
  `logits` is glossary-pinned even though E1 slices to the 20 canonical AAs — same as `esm2`,
  so not a defect.) All fields use field-level `Field(...)` (no `Optional[Annotated[..., Field]]`
  drop).
- **Resource specs** are consistent across `config.py`, README (both tables), and MODEL.md
  (150m→T4/8GB/fp16; 300m→L4/16GB/bf16; 600m→L4/24GB/bf16).
- **License:** per-model Apache-2.0 `LICENSE` present and consistent with `sources.yaml`
  (`type: Apache-2.0`, HF URL). No raw third-party PDF shipped in-repo (PDF referenced only via
  R2 path).
- **log_prob method:** computes an unmasked single-pass sum of canonical-AA log-softmax — the
  same approach as `esm2.log_prob`; house-consistent and glossary-blessed as
  "pseudo-log-likelihood." (Noting only that it is a wild-type-marginal rather than a true
  per-position masked PLL; not flagged because it matches the reference model.)
- **Response sub-models** (`LayerEmbedding`/`LayerPerTokenEmbeddings`) correctly inherit
  `ResponseModel` (e1 fixes the `RequestModel` inheritance that `esm2` still has).

## Verification

Adversarial re-check of the two high-severity findings against the actual source:

- **Finding 1 — Docs call E1 "Encrypted" (it is an Encoder model): REAL.**
  Confirmed verbatim in shipped public docs: `models/e1/README.md:3`
  ("Encrypted protein language model with retrieval-augmented inference…"),
  `models/e1/MODEL.md:7` ("E1 is an encrypted protein language model…"; the
  finding's `:6` cite is the section header, the offending sentence is line 7),
  and `models/e1/sources.yaml:6` ("Encrypted protein embedding model by
  Profluent/Synthyra"). The model is an encoder, not encrypted: the paper title
  in `sources.yaml:17` is "E1: Retrieval-Augmented Protein Encoder Models",
  weights are `Synthyra/Profluent-E1-*`, and the README itself elsewhere calls it
  a "Transformer encoder" (`README.md:15`) and "encoder-only model"
  (`README.md:42`). The "encrypted" wording is a clear, repeated factual error.

- **Finding 2 — encode() silently drops out-of-bounds repr_layers instead of raising: REAL.**
  `models/e1/app.py:337-340` filters with `if 0 <= pos_lyr < n_layers:` and only
  appends in-range layers; out-of-range indices are silently dropped, so
  `repr_layers=[25]` on the 150m variant (21 hidden states) yields
  `layers_to_use=[]` and `embeddings=[]` with no error. Nothing upstream catches
  it: `E1EncodeRequestParams.repr_layers` (`schema.py:49-52`) is a bare
  `list[int]` with no bounds validation. The house pattern in
  `models/esm2/app.py:244-248` explicitly validates the range and raises
  `ValidationError400` on out-of-bounds layers — so this is both a UX footgun and
  a uniformity break versus the reference model.
