# Review — `models/omni_dna/` (Round 1)

## Summary

Omni-DNA is a single-deployed-variant (1B) DNA foundation model exposing two actions, `encode`
and `log_prob`, built on the OLMo/`ai2-olmo` causal-LM architecture. The plumbing is largely
house-standard: it uses `parse_variant`, `setup_download_layer`/`setup_source_layer`,
`ModelMixinSnap`, `@biolm_model_class`, `modal_endpoint`, `get_logger` (no `print`), the canonical
`r2_then_hf` acquisition with `huggingface_hub` correctly listed in `extra_pip_packages`, typed
Pydantic request/response schemas with rendered field descriptions, and a `TestSuite` with
integration + deployment cases. License is genuinely Apache-2.0 and matches `sources.yaml`. No
secret / `biolm-modal` / `.planning` leakage was found in shipped files.

The problems are concentrated in (1) the **model-loading path** — `from_config` +
`load_state_dict(strict=False)` can silently drop mismatched weights and the golden-fixture tests
cannot detect it (both golden and actual come from the same deterministic path); (2) the **encode
response shape**, which diverges from sibling DNA model `dnabert2` and wraps a single embedding in a
length-1 list of objects; and (3) several **knowledge-graph accuracy/consistency defects** — a
fabricated paper title across three of the five docs, a dangling `nt` model cross-reference,
contradictory max-length documentation, a reviewer meta-note shipped inside the LICENSE NOTICE, and
a slug mismatch. None rise to a hard 🔴 (the model deploys and its integration tests pass), but the
loading path and the doc inaccuracies should be fixed before launch.

Compared against `models/esm2/` (reference) and `models/dnabert2/` (closest sibling — same
`*PredictLogProb*` naming, same `validate_dna_unambiguous`, same DNA-encode role).

## 🟠 Should-fix

### 1. Model loading uses `from_config` + `load_state_dict(strict=False)` — silent weight-drop risk and diverges from the house pattern
- **Category:** Correctness / Convention
- **Location:** `models/omni_dna/app.py:131-141`
- **Detail:** The model is built from a randomly-initialized config and then overlaid with weights:
  ```python
  self.model = AutoModelForCausalLM.from_config(config, trust_remote_code=True)
  ...
  state_dict = load_file(str(safetensors_path))
  self.model.load_state_dict(state_dict, strict=False)
  ```
  `strict=False` means any key mismatch (a `model.` prefix difference, tied embeddings, a renamed
  remote-code parameter) is **silently ignored**, leaving those tensors at their random init. This
  bypasses HF's `from_pretrained` key-remapping/tying logic. The sibling `dnabert2` does the safe
  thing — `AutoModelForMaskedLM.from_pretrained(snapshot_path, trust_remote_code=True)`
  (`models/dnabert2/app.py:119-121`) — so this is also a plumbing divergence. Worse, the golden
  fixtures are deterministic (seed 42) and were generated by this same path, so the integration
  tests would pass even if a block of weights never loaded; the "VERIFIED" claim in `README.md` does
  not actually guarantee a clean load.
- **Suggested fix:** Prefer `AutoModelForCausalLM.from_pretrained(snapshot_dir, trust_remote_code=True)`
  like `dnabert2`. If `from_config` is genuinely required for GPU-snapshot reasons, capture the
  result and fail loudly: `res = self.model.load_state_dict(state_dict, strict=True)` (or
  `strict=False` then assert `not res.missing_keys and not res.unexpected_keys`, logging both).
  (The "Load model directly on GPU" docstring is also inaccurate — `from_config` builds on CPU, then
  `.to(device)` moves it; same minor wording bug exists in `dnabert2`.)

### 2. `encode` response shape diverges from siblings and over-wraps a single embedding
- **Category:** Consistency / Simplicity
- **Location:** `models/omni_dna/schema.py:108-130`, `models/omni_dna/app.py:208-219`
- **Detail:** `encode` returns `results[i].mean = [ {embedding: [...]} ]` and `.last = [ {...} ]` —
  an `Optional[list[OmniDNAEncodeResponseEmbedding]]` where the list is **always length 1** (see
  `app.py:212-218`, which appends exactly one object). The sibling DNA encoder returns a flat
  `embedding: list[float]` (`dnabert2/schema.py:78-81`); `esm2` returns `embeddings:
  list[LayerEmbedding]` where the list is justified by multiple requested layers — a justification
  Omni-DNA does not have (one hidden layer, one pooling each). So a consumer must dig through
  `results[0].mean[0].embedding` vs `results[0].embedding` on dnabert2 for the same concept. The
  north-star ("the diff between any two models is the science, not the plumbing") is violated for the
  most common action. Also `OmniDNAEncodeResponseEmbedding` subclasses `RequestModel` for a response
  object (matches the `esm2` quirk, so not separately flagged).
- **Suggested fix:** Drop the wrapper object and the length-1 list: make `mean: Optional[list[float]]`
  and `last: Optional[list[float]]` (Field descriptions kept). If keeping a single canonical
  `embeddings` field with the pooling expressed as a tag is preferred, align with the family
  convention — but at minimum remove the gratuitous list-of-one nesting.

### 3. Fabricated / inconsistent paper title and authorship across three knowledge-graph files
- **Category:** Docs / Knowledge graph accuracy
- **Location:** `models/omni_dna/README.md:217`, `models/omni_dna/MODEL.md:63`, `models/omni_dna/BIOLOGY.md:52`
- **Detail:** `sources.yaml:16` correctly records the real arXiv 2502.03499 title — *"Omni-DNA: A
  Unified Genomic Foundation Model for Cross-Modal and Multi-Task Learning"* — with all seven
  authors. README/MODEL/BIOLOGY instead cite a **different, non-existent title** *"Omni-DNA: A
  Unified Multi-Task Framework for DNA Foundation Models"* and attribute it to "Li Z." alone. This is
  an outright citation inaccuracy in shipped public docs and contradicts the model's own
  `sources.yaml`.
- **Suggested fix:** Replace every occurrence with the canonical title and the full author list (or
  "Li et al., 2025") exactly as in `sources.yaml`; cite arXiv:2502.03499.

### 4. `comparison.yaml` references model `nt` that does not exist in `models/`
- **Category:** Knowledge graph consistency
- **Location:** `models/omni_dna/comparison.yaml:53-55` (and prose at lines 27, 35, 41, 43)
- **Detail:** The structured `alternatives:` block contains `- model: "nt"`, and prose references
  `nt` / `nt-v2-250m`. There is no `nt`, `nucleotide_transformer`, or similar directory under
  `models/` (verified: `dnabert2`, `evo`, `evo2`, `dna_chisel`, `esm2` all exist; `nt` does not).
  The file's own header states *"All referenced model slugs must exist in models/"*; a catalog UI
  iterating `alternatives[].model` would produce a dead link.
- **Suggested fix:** Remove the `nt` `alternatives` entry (and the `nt-v2-250m` prose) or replace
  with a model that exists, until/unless a Nucleotide-Transformer model is added.

### 5. Max-sequence-length documentation is internally inconsistent and contradicts the schema
- **Category:** Docs / Correctness
- **Location:** `models/omni_dna/README.md:20-21, 55-56, 197`; `models/omni_dna/MODEL.md:23, 57`
- **Detail:** The actual enforced limit is **2,048 characters** (`schema.py:58`/`87`:
  `max_length=OmniDNAParams.max_sequence_len` on the `str` field, i.e. nucleotides, not tokens). The
  docs variously claim: "Max sequence length 2,048 BPE tokens" (README arch table), "truncated at
  2,048 BPE tokens (approximately 4,000–8,000 nucleotides)" (README L55, MODEL L57) — which is
  unreachable since input is hard-capped at 2,048 chars — and, contradicting both, README Notes L197:
  *"trained on sequences of up to about 250 BPE tokens, it can handle sequences of roughly 512
  tokens."* The encode request table ("1–2048 nt", README L68) is the only accurate statement.
- **Suggested fix:** State a single consistent limit: input is capped at 2,048 nucleotides
  (characters); the tokenizer truncates at 2,048 BPE tokens but that bound is never hit given the
  character cap. Delete the stale "250/512 token" note (looks copied from another model).

### 6. LICENSE NOTICE ships a reviewer meta-instruction and a likely-wrong affiliation
- **Category:** Licensing / Review residue
- **Location:** `models/omni_dna/LICENSE:174-175, 180-182`
- **Detail:** The NOTICE block ends with *"…attribution is to 'The Omni-DNA Authors' **(flag for
  reviewer to verify)**."* — an internal review TODO left inside a file meant to ship publicly. It
  also states the authors are from "Imperial College London, MIT, and collaborators"; the paper's
  affiliations are Imperial / Microsoft Research / Toronto (no MIT), so "MIT" appears hallucinated.
- **Suggested fix:** Resolve the attribution now (the upstream HF repo / paper lists the authors;
  cite them or "The Omni-DNA Authors" cleanly), remove the "(flag for reviewer to verify)"
  parenthetical, and correct or drop the affiliation list.

## 🟡 Nits

### 7. `last` pooling and `log_prob` assume right-padding (no `padding_side` assertion) — *medium confidence*
- **Category:** Correctness (latent)
- **Location:** `models/omni_dna/app.py:198-205` (last-token gather), `app.py:268-280` (log-prob shift)
- **Detail:** Last-token pooling computes `last_indices = attn_mask.sum(dim=1) - 1`, and `log_prob`
  uses the standard `[:, 1:]` / `[:, :-1]` shift with `attention_mask[:, 1:]`. Both are correct only
  if the tokenizer pads on the **right**. If the OLMo tokenizer defaults to left-padding (common for
  causal LMs), the "last" index lands inside the padded region and the log-prob shift counts a
  pad→real-token transition. With `batch_size=2` and unequal lengths this would silently mis-score
  the second item — and the golden-fixture tests can't catch it (same deterministic path). Mean
  pooling is unaffected. I could not confirm the tokenizer's padding side without loading the model
  (Modal cost-discipline), hence 🟡.
- **Suggested fix:** Assert `self.tokenizer.padding_side == "right"` in `setup_model` (or set it
  explicitly), so a wrong default fails fast rather than silently.

### 8. README states the wrong container base image
- **Category:** Docs
- **Location:** `models/omni_dna/README.md:189`
- **Detail:** "Built from `pytorch/pytorch:2.3.1-cuda11.8-cudnn8-runtime`" — but `app.py:46` uses
  `pytorch/pytorch:2.6.0-cuda12.4-cudnn9-runtime` (the `2.3.1` value is dnabert2's). Could mislead a
  contributor reproducing the CUDA environment.
- **Suggested fix:** Update to `2.6.0-cuda12.4-cudnn9-runtime`.

### 9. `comparison.yaml` `model_slug` uses the directory name, not the public slug
- **Category:** Consistency
- **Location:** `models/omni_dna/comparison.yaml:10`
- **Detail:** `model_slug: "omni_dna"` (underscore) while `config.py` `base_model_slug` and
  `sources.yaml` both use `omni-dna` (hyphen). The repo is split on this (e.g. `dna_chisel`,
  `esm_if1` have the same hyphen/underscore mismatch; `msa-transformer`, `thermompnn-d` use the
  hyphenated slug in comparison.yaml), so this is a pre-existing inconsistency the model inherited,
  but it still fails the "slug matches config" check within this model's own files.
- **Suggested fix:** Use `omni-dna` to match `config.py`/`sources.yaml` (and ideally standardize
  repo-wide in the W5 pass).

### 10. Shipped placeholders: `TODO` markers and `snapshot_r2: pending`
- **Category:** Knowledge graph completeness (largely repo-wide)
- **Location:** `models/omni_dna/schema.py:23` (`# TODO: test how long sequences can be…`),
  `README.md:149`, `MODEL.md:33, 63` (`<!-- TODO … -->`), `sources.yaml:36` (`snapshot_r2: pending`)
- **Detail:** The rubric flags stray `TODO`/`pending`. The schema `TODO` is copy-pasted from
  `dnabert2/schema.py:18`; the `<!-- TODO -->` HTML comments and `pending` placeholders appear across
  many models, so these are repo-wide patterns rather than Omni-DNA-specific — noting for completeness
  and so the global cleanup pass catches them.
- **Suggested fix:** Resolve or remove the markers; archive the HF snapshot to R2 and replace
  `pending` with the path (or drop the field if not used).

### 11. MODEL.md claims DNABERT-2 has "no BioLM integration" — false
- **Category:** Docs accuracy
- **Location:** `models/omni_dna/MODEL.md:81`
- **Detail:** The comparison table lists DNABERT-2's disadvantage as "Shorter context; no BioLM
  integration", but `models/dnabert2/` is a sibling model in this very repo.
- **Suggested fix:** Remove the "no BioLM integration" clause.

### 12. `__main__` usage docstring references the internal "QA/prod" environment
- **Category:** Internal-reference (minor / global)
- **Location:** `models/omni_dna/app.py:295`
- **Detail:** "Force deploy in QA/prod:". The rubric lists the internal `qa` env as a leak, but the
  reference model `esm2/app.py:484` ("Force deploy to 'qa' or 'main' environment") and `dnabert2`
  carry the identical wording — this is a repo-wide convention best fixed globally, not a per-model
  blocker.
- **Suggested fix:** Generalize to environment-neutral wording (e.g. "Force deploy:") across models.

## Verification

Adversarial re-check of the six HIGH-severity findings against the actual files. Verdicts:

1. **Model loading `from_config` + `load_state_dict(strict=False)` — REAL.** app.py:131-141 literally does
   `AutoConfig.from_pretrained` → `AutoModelForCausalLM.from_config(config, trust_remote_code=True)` →
   `load_file(...)` → `self.model.load_state_dict(state_dict, strict=False)`. `strict=False` does silently
   ignore key mismatches by design, and the sibling dnabert2 app.py:119-121 uses the safe
   `AutoModelForMaskedLM.from_pretrained(snapshot_path, trust_remote_code=True)` — so both the silent-drop
   risk and the plumbing divergence are demonstrable. (Risk is potential, not a confirmed live bug, but the
   code is exactly as described.)

2. **encode over-wraps a single embedding in a length-1 list — REAL.** schema.py:123-130 types `mean`/`last`
   as `Optional[list[OmniDNAEncodeResponseEmbedding]]`; app.py:211-218 appends exactly one element to each,
   so the list is always length 1. dnabert2 schema.py:79 returns flat `embedding: list[float]`; esm2
   schema.py:155 returns `embeddings: list[LayerEmbedding]` justified by multiple `repr_layers` (a
   justification Omni-DNA lacks). Cross-model shape divergence confirmed; `OmniDNAEncodeResponseEmbedding`
   subclassing RequestModel (line 108) mirrors esm2's LayerEmbedding (line 129), correctly not re-flagged.

3. **Fabricated/inconsistent paper title & authorship — REAL.** sources.yaml:16 holds the real arXiv
   2502.03499 title "...A Unified Genomic Foundation Model for Cross-Modal and Multi-Task Learning" + 7
   authors; README.md:217 and BIOLOGY.md:52 instead cite "...A Unified Multi-Task Framework for DNA
   Foundation Models" attributed to "Li Z." alone, and MODEL.md:63 repeats that wrong title as "Li et al."
   Title contradiction across all three shipped docs vs the model's own sources.yaml is confirmed (minor
   nit: MODEL.md says "Li et al.", not "Li Z." — does not change the verdict).

4. **comparison.yaml references nonexistent model `nt` — REAL.** comparison.yaml:53 `- model: "nt"` in the
   structured `alternatives` block, plus prose at 27/35/41/43 (nt / NT / nt-v2-250m). `ls models/` shows no
   `nt`/`nucleotide_transformer` dir (dnabert2, evo, evo2, dna_chisel, esm2 exist; nt does not), while the
   file's own header line 8 mandates "All referenced model slugs must exist in models/".

5. **Max-seq-length docs inconsistent & contradict the 2048-char schema cap — REAL.** schema.py:58/87 set
   `max_length=OmniDNAParams.max_sequence_len` (=2048, schema.py:24) on the `sequence: str` field, i.e. a
   2,048-*character* (nucleotide) cap. README.md:20 "2,048 BPE tokens", README.md:55 "truncated at 2,048
   BPE tokens (~4,000–8,000 nucleotides)" (unreachable under a 2,048-char input cap), and README.md:197
   "~250 BPE tokens, can handle ~512 tokens" all conflict with each other and the schema; MODEL.md:23/57
   repeat the BPE-token framing. Only the encode request table (README.md:68 "1–2048 nt") is accurate.

6. **LICENSE NOTICE ships a reviewer meta-instruction + likely-wrong affiliation — REAL.** LICENSE:181-182
   ends "...attribution is to 'The Omni-DNA Authors' (flag for reviewer to verify)." — an internal review
   TODO inside a public-shipping file (incontrovertible). LICENSE:175 lists "Imperial College London, MIT,
   and collaborators"; the author roster (sources.yaml:21-27: Imperial / Toronto / Microsoft Research) shows
   no MIT, so "MIT" is plausibly hallucinated — that sub-claim relies on external paper-affiliation knowledge
   (uncertain on its own), but the meta-instruction defect alone makes the finding real.

**Summary: all 6 findings REAL.** Each is concretely demonstrable in the cited files; no refutations found.
